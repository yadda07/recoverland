RecoverLand : Formats

Formats supportes

RecoverLand fonctionne avec tous les formats editables de QGIS, mais tous les formats n'offrent pas le meme niveau de fiabilite. Cette page decrit precisement ce qui fonctionne, ce qui fonctionne partiellement, et ce qui ne fonctionne pas.

Capture

"Oui" signifie que RecoverLand enregistre les modifications faites sur ce format. L'enregistrement est identique quel que soit le format : attributs, geometrie, type d'operation.

Restauration

"Oui" signifie que RecoverLand peut re-appliquer un etat anterieur sur la couche. "Oui (risque)" signifie que la restauration est techniquement possible mais que le resultat peut etre incorrect si les identifiants d'entite ont change entre temps.

Identification

La force d'identification determine la fiabilite avec laquelle RecoverLand retrouve la bonne entite au moment de la restauration :

  • STRONG : Cle primaire stable. L'entite est retrouvee de maniere certaine.
  • MEDIUM : FID generalement stable. Correct dans la plupart des cas, mais fragile apres certaines operations sur le fichier.
  • WEAK : FID = numero de ligne. Change des qu'une ligne est ajoutee ou supprimee.
  • NONE : Pas d'identifiant stable. Restauration impossible ou non pertinente.

GeoPackage (.gpkg)

STRONG

Le GeoPackage est le format recommande pour RecoverLand. C'est un fichier SQLite avec des extensions spatiales standardisees par l'OGC.

Pourquoi c'est le meilleur choix

  • Cle primaire auto-incrementee (fid) qui ne reutilise jamais les valeurs supprimees
  • Un seul fichier (pas de .shx, .dbf, .prj a gerer)
  • Support multi-couches dans un seul fichier
  • Support des types de donnees riches (date, datetime, entier 64 bits, blob)
  • Pas de limite de 254 caracteres sur les noms de champs (contrairement au shapefile)
Recommandation : Si vous commencez un nouveau projet et que vous avez le choix du format, utilisez GeoPackage. C'est le format qui offre la meilleure fiabilite avec RecoverLand.

Shapefile (.shp)

MEDIUM

Le shapefile est le format historique du SIG. RecoverLand le supporte entierement pour la capture et la restauration, avec une reserve sur la stabilite des FID.

Ce qui fonctionne

  • Capture de toutes les modifications (attributs, geometries, suppressions, insertions)
  • Restauration correcte tant que le fichier n'a pas ete re-exporte ou compacte

Ce qui peut echouer

FID instable : Le FID d'un shapefile est le numero de ligne dans le fichier .dbf. Si le shapefile est re-exporte, compacte (Pack), ou si des entites sont supprimees puis le fichier reorganise, les FID changent. La restauration risque de modifier la mauvaise entite.

Quand ca pose probleme concretement

  • Vous exportez le shapefile avec "Enregistrer sous..." dans un nouveau fichier
  • Un script externe reorganise ou compacte le .dbf
  • Vous supprimez des entites puis faites un "Pack" du shapefile dans un autre outil

Si vous editez le shapefile uniquement dans QGIS sans le re-exporter, les FID restent stables et la restauration fonctionne correctement.

GeoJSON (.geojson)

MEDIUM

Meme comportement que le shapefile. Les FID sont generalement stables tant que le fichier est edite uniquement dans QGIS. Un re-export ou une edition externe peut changer les FID.

FlatGeobuf (.fgb)

STRONG

FlatGeobuf utilise des identifiants stables similaires au GeoPackage. C'est un bon choix pour les fichiers volumineux grace a son index spatial integre.

CSV / Texte delimitie

WEAK

Le CSV n'a pas de notion de cle primaire. Le FID est le numero de ligne dans le fichier.

Restauration tres risquee : Ajouter ou supprimer une seule ligne decale tous les FID suivants. RecoverLand capture les modifications, mais la restauration a de grandes chances de modifier la mauvaise ligne si le fichier a change depuis.

Usage recommande

Utilisez RecoverLand sur un CSV uniquement pour consulter l'historique (voir ce qui a change et quand). Evitez la restauration automatique sur ce format. Si vous devez restaurer, exportez le CSV en GeoPackage d'abord.

PostgreSQL / PostGIS

STRONG

Les tables PostgreSQL ont une cle primaire explicite (generalement gid ou id). RecoverLand enregistre cette cle dans feature_identity_json et l'utilise pour la restauration.

Deux modes : RecoverLand supporte PostgreSQL via le backend SQLite local (capture automatique) et via le backend legacy (connexion directe). Le mode local est recommande car il ne depend pas de la disponibilite du serveur pour consulter l'historique.

SpatiaLite

STRONG

Similaire au GeoPackage : fichier SQLite avec extensions spatiales, cle primaire auto-incrementee stable. Entierement supporte.

MS SQL Server

STRONG

Cle primaire explicite, identification forte. Entierement supporte pour la capture et la restauration.

Oracle Spatial

STRONG

Cle primaire explicite, identification forte. Entierement supporte pour la capture et la restauration.

Couche memoire

NONE

Les couches memoire n'ont pas de fichier source. Elles existent uniquement dans la session QGIS courante et disparaissent a la fermeture.

RecoverLand capture les modifications a titre informatif (vous pouvez voir l'historique), mais la restauration est impossible car il n'y a pas de cible persistante ou re-appliquer les donnees.

WFS (Web Feature Service)

MEDIUM

Les couches WFS-T (transactionnelles) sont supportees pour la capture et la restauration. La fiabilite depend du serveur WFS et de la stabilite des identifiants qu'il fournit.

Dependance reseau : La restauration necessite que le serveur WFS soit accessible au moment de la restauration. Si le serveur est hors ligne, la restauration echoue.

Couche virtuelle

REFUSE

Les couches virtuelles sont des vues derivees d'autres couches. RecoverLand les refuse completement : pas de capture, pas de restauration. Modifier une couche virtuelle n'a pas de sens car elle est recalculee a partir de ses sources.

Recommandation de format

Si vous avez le choix du format, utilisez GeoPackage. C'est un fichier unique, portable, avec des identifiants stables. RecoverLand fonctionnera de maniere optimale avec ce format.
