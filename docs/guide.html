RecoverLand : Guide pratique

Guide utilisateur

Ce guide couvre l'utilisation quotidienne de RecoverLand : de l'installation a la restauration de donnees. Chaque etape est illustree pour que tout utilisateur QGIS puisse suivre, meme sans connaissance technique du plugin.

Installation

1
Ouvrir le gestionnaire d'extensions

Dans QGIS, ouvrez Extensions → Installer/Gerer les extensions.

2
Installer RecoverLand

Recherchez "RecoverLand" dans la barre de recherche. Cliquez sur Installer.

Si le plugin n'est pas dans le depot officiel, copiez le dossier recoverland dans :

C:\Users\[votre_nom]\AppData\Roaming\QGIS\QGIS3\profiles\default\python\plugins\

Puis redemarrez QGIS.

3
Verifier l'activation

Une icone RecoverLand apparait dans la barre d'outils QGIS. Si elle n'apparait pas, verifiez dans Extensions → Installer/Gerer les extensions → Installees que RecoverLand est coche.

Configuration requise : QGIS 3.44 minimum. Aucune dependance externe. Le plugin utilise uniquement Python standard et SQLite, tous deux inclus dans QGIS.

Premier lancement

1
Ouvrir un projet QGIS avec des couches editables

RecoverLand a besoin d'au moins une couche vecteur editable pour fonctionner. Si vous n'avez pas de projet, creez-en un et ajoutez un GeoPackage ou un Shapefile.

2
Cliquer sur l'icone RecoverLand

La fenetre RecoverLand s'ouvre. En haut, la barre d'information affiche :

  • Journal local : nom et taille du journal SQLite
  • Compteurs : nombre total d'evenements, updates, suppressions, insertions dans le scope courant
  • Toggle d'enregistrement : active ou desactive la capture des modifications
3
Verifier que l'enregistrement est actif

Le toggle en haut a droite doit indiquer "Enregistrement actif" en vert. Si c'est rouge, activez-le.

Des ce moment, RecoverLand enregistre silencieusement chaque modification que vous faites sur les couches surveillees. Vous n'avez rien d'autre a configurer.

Vue d'ensemble de l'interface

La barre d'information

La barre en haut de la fenetre affiche un resume en temps reel du journal. Les compteurs se mettent a jour quand vous changez les filtres de selection ou de periode. Cliquer sur une tuile (Total, Updates, Suppr., Ajouts) bascule le filtre d'operation correspondant.

Suivi des modifications

Par defaut, RecoverLand surveille toutes les couches editables du projet. Vous pouvez restreindre la surveillance a certaines couches dans la section Couches surveillees.

Activer/desactiver l'enregistrement

Le toggle en haut a droite de la barre d'information controle globalement la capture. Quand il est desactive, aucune modification n'est enregistree, meme si vous editez des couches.

Desactiver = aucune protection. Si vous desactivez l'enregistrement, les modifications faites pendant cette periode sont perdues definitivement. RecoverLand ne peut pas les retrouver apres coup.

Choisir les couches surveillees

Ouvrez la section Couches surveillees (repliee par defaut). Chaque couche vecteur chargee dans le projet apparait avec une case a cocher. Decochez une couche pour arreter de surveiller ses modifications.

Rechercher des modifications

1
Choisir la couche et l'operation

Dans la section Selection :

  • Couche : "Toutes les couches sauvegardees" ou une couche specifique
  • Operation : Toutes, UPDATE, DELETE, ou INSERT
2
Definir la periode

Utilisez les champs Date debut et Date fin, ou les raccourcis (10 min, 30 min, 1 heure, 1 jour) pour definir rapidement la fenetre de recherche.

3
Cliquer sur Recover

Le bouton Recover lance la recherche dans le journal SQLite. Une barre de progression s'affiche pendant le traitement. Les resultats apparaissent dans le tableau.

Lire les resultats

Le tableau affiche chaque evenement avec des colonnes fixes et des colonnes dynamiques.

Colonnes fixes

ColonneContenu
OperationINSERT, UPDATE ou DELETE
CoucheNom de la couche au moment de la modification
DateDate et heure UTC de la modification
UtilisateurNom de l'utilisateur qui a effectue la modification

Colonnes dynamiques (attributs)

Les colonnes suivantes montrent les attributs de l'entite au moment de la modification. Les cellules sont colorees pour indiquer le type de changement :

CouleurSignification
Bleu clairValeur modifiee (ancienne valeur affichee)
Rouge clairValeur videe (etait remplie, maintenant vide)
Vert clairValeur ajoutee (etait vide, maintenant remplie)
Orange clairGeometrie modifiee
Filtre texte : Un champ de recherche au-dessus du tableau permet de filtrer les lignes par mot-cle. La recherche porte sur toutes les colonnes visibles.

Restaurer des donnees

1
Selectionner les lignes a restaurer

Cochez les lignes du tableau que vous souhaitez restaurer. Utilisez Tout selectionner / Tout deselectionner pour les operations en lot.

2
Cliquer sur Restore

Un message de confirmation apparait avec le nombre d'entites a restaurer. Apres confirmation, RecoverLand :

  • Verifie que la couche cible existe et est editable
  • Compare le schema actuel avec le schema sauvegarde
  • Restaure les attributs et la geometrie a leur etat anterieur
  • En cas d'erreur sur une entite, les autres entites du lot sont quand meme traitees
La restauration modifie les donnees. Elle ecrase les valeurs actuelles par les valeurs enregistrees dans le journal. Assurez-vous de selectionner les bonnes lignes avant de confirmer.
Restauration sur shapefile recompacte : Si un shapefile a ete re-exporte ou compacte entre le moment de la capture et la restauration, les FID ont pu changer. La restauration risque de modifier la mauvaise entite. Privilegiez le GeoPackage pour eviter ce risque.

Avertissement de derive de schema

Si la structure d'une couche a change entre la capture et la restauration (champ ajoute, supprime ou renomme), RecoverLand affiche un avertissement avant de proceder.

Ce qui se passe concretement

1
Vous lancez une restauration

RecoverLand compare le schema enregistre dans l'evenement (field_schema_json) avec le schema actuel de la couche.

2
Differences detectees

Le rapport indique :

  • Champs compatibles : restauration normale
  • Champs manquants : existaient a la capture mais ont ete supprimes. Valeurs ignorees.
  • Champs nouveaux : n'existaient pas a la capture. Laisses a leur valeur par defaut.
  • Champs incompatibles : le type a change (ex: texte vers entier). Tentative de conversion.
Bonne pratique : Evitez de modifier la structure d'une couche si vous prevoyez de restaurer des donnees anciennes. Si c'est inevitable, verifiez le rapport de compatibilite avant de confirmer.

Gerer la retention du journal

Le journal SQLite grossit au fil du temps. RecoverLand offre des outils pour maitriser sa taille.

Purge par anciennete

Supprimez les evenements plus anciens qu'un seuil : 30 jours, 90 jours, 365 jours. Les evenements purges sont definitivement supprimes.

Purge par session

Supprimez tous les evenements d'une session d'edition specifique. Utile pour nettoyer les sessions de test ou les imports massifs.

Compactage (VACUUM)

Apres une purge, la taille physique du fichier SQLite ne diminue pas automatiquement. Le compactage VACUUM recupere l'espace disque. Cette operation peut prendre quelques secondes sur un journal volumineux.

Purge = suppression definitive. Une fois purges, les evenements ne sont plus recuperables. Assurez-vous que l'historique purge n'est plus necessaire.

Surveillance de la taille

La barre d'information en haut de la fenetre RecoverLand affiche en permanence la taille du journal et le nombre d'evenements. Surveillez ces indicateurs pour savoir quand une purge est necessaire.

Mode PostgreSQL (legacy)

Si votre projet utilise des couches PostgreSQL avec des triggers d'audit installes cote serveur (schema suppresions), RecoverLand detecte automatiquement ce backend et l'utilise pour ces couches.

Coexistence des deux modes

Couche GeoPackage → SQLite local
Couche PostGIS + triggers → PostgreSQL legacy

Les deux modes coexistent dans le meme projet. Le BackendRouter decide du backend pour chaque couche. Vous n'avez rien a configurer : la detection est automatique.

Recommandation : Le mode SQLite local est recommande car il ne depend pas de la disponibilite du serveur pour consulter l'historique. Le mode PostgreSQL est conserve pour la compatibilite avec les installations existantes.

Scenario : Annuler une erreur

Vous venez de modifier par erreur 50 attributs sur une couche de vegetation.

Ouvrir RecoverLand → Couche = vegetation → Operation = UPDATE → Periode = 10 min → Recover

Les 50 modifications apparaissent. Selectionnez-les toutes, puis cliquez Restore. Les attributs reviennent a leur etat precedent.

Scenario : Comparer avant/apres

Vous voulez voir ce qui a change sur une couche depuis hier.

Couche = parcelles → Operation = Toutes → Periode = 1 jour → Recover

Le tableau montre toutes les modifications. Les cellules colorees indiquent exactement quels attributs ont change et quelles etaient les anciennes valeurs.

Scenario : Retrouver une entite supprimee

Un collegue a supprime par erreur une entite importante.

Couche = batiments → Operation = DELETE → Periode = 1 jour → Recover

L'entite supprimee apparait avec tous ses attributs et sa geometrie au moment de la suppression. Selectionnez-la et cliquez Restore pour la re-inserer dans la couche.

L'entite restauree est inseree comme une nouvelle entite avec un nouveau FID. Les attributs et la geometrie sont identiques a l'etat sauvegarde.
