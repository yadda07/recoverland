Ce que RecoverLand fait

  • Capture chaque modification validee (commit) sur les couches editables surveillees
  • Stockage delta : seuls les attributs reellement modifies sont enregistres (reduction 10-50x du volume)
  • Tampon d'edition intelligent : fusionne les modifications multiples sur une meme entite en un seul evenement net
  • Filtre automatique des champs d'audit de la couche (date_modif, updated_by, etc.)
  • Recherche paginee dans l'historique par couche, operation, date, utilisateur
  • Restauration selective avec verification de compatibilite de schema
  • Detection de derive de schema entre l'etat sauvegarde et l'etat actuel
  • Mode dual backend : SQLite local (automatique) + PostgreSQL legacy (si disponible)
  • Ecriture asynchrone par lots dans un thread dedie (zero gel UI)
  • Verification d'integrite du journal au demarrage (WAL checkpoint, recovery d'evenements en attente)
  • Identification utilisateur automatique (config plugin, variable env, os.getlogin, profil QGIS)
  • Fonctionne sur tous les formats editables : GeoPackage, Shapefile, PostGIS, SpatiaLite, GeoJSON, CSV

Ce que RecoverLand ne fait pas

  • Ne remplace pas un systeme de versionnement (Git, GeoGig)
  • Ne sauvegarde pas les fichiers eux-memes, seulement les modifications d'entites
  • Ne fonctionne pas sur les couches non editables (WMS, rasters, couches virtuelles)
  • Ne capture pas les modifications faites en dehors de QGIS (edition directe du fichier)
  • Ne synchronise pas entre plusieurs postes de travail
  • Ne garantit pas la restauration sur les formats a identification faible (CSV, shapefiles recompactes)
Honnetete : RecoverLand est un filet de securite local, pas une solution de versionnement distribuee. Il protege contre les erreurs de manipulation dans QGIS, pas contre la perte du fichier source lui-meme.

Journal SQLite

Le journal est un fichier SQLite unique par projet QGIS. Il utilise le mode WAL (Write-Ahead Logging) pour permettre des lectures et ecritures simultanees sans blocage.

Emplacement du journal

SituationEmplacement
Projet sauvegarde dans C:/projets/chantier.qgzC:/projets/.recoverland/audit.db
Projet non sauvegarde[profil QGIS]/recoverland/audit/audit.db

Capture automatique

La capture est transparente. L'utilisateur edite ses couches normalement dans QGIS. RecoverLand intercepte les signaux d'edition au moment du commit.

Utilisateur edite → beforeCommitChanges → Snapshot des entites → afterCommitChanges → Ecriture journal

Quand la capture se declenche

Action utilisateurSignal QGISResultat dans le journal
Modifier un attribut et sauvegarderbeforeCommitChanges + afterCommitChangesEvenement UPDATE avec l'ancien etat
Deplacer une geometrie et sauvegarderIdemEvenement UPDATE avec l'ancienne geometrie
Supprimer une entite et sauvegarderIdemEvenement DELETE avec l'etat complet avant suppression
Ajouter une entite et sauvegarderIdemEvenement INSERT avec l'etat apres ajout
Annuler l'edition (rollback)afterRollBackRien (les snapshots en attente sont supprimes)
Modification hors QGIS : Si un fichier est modifie directement (avec un editeur de texte, un script Python externe, ou un autre logiciel), RecoverLand ne le detecte pas. Seules les modifications faites via l'interface d'edition de QGIS sont capturees.

Tampon d'edition intelligent

Le tampon d'edition (EditSessionBuffer) est la memoire vive de RecoverLand. Il capture l'etat initial de chaque entite modifiee au debut de la session et calcule l'effet net reel au moment du commit.

Pourquoi un tampon ?

Un utilisateur peut modifier la meme entite plusieurs fois avant de sauvegarder. Sans tampon, chaque micro-modification genererait un evenement distinct. Le tampon fusionne tout en un seul evenement representant le changement reel entre l'etat initial et l'etat final.

Modifier attribut A → Modifier attribut B → Remettre A a sa valeur initiale → Effet net : seul B a change

Cas speciaux geres par le tampon

ScenarioResultat dans le journal
Modifier puis remettre a l'identiqueAucun evenement (non-changement detecte)
Inserer puis supprimer avant commitAucun evenement (effet net nul)
Modifier un attribut + deplacer la geometrieUn seul evenement UPDATE avec le delta attribut + ancienne geometrie
Rollback de la sessionTampon purge, rien n'est ecrit

Filtrage des champs d'audit

Les champs de metadonnees de la couche (date_modif, updated_by, updated_at, etc.) sont automatiquement exclus du delta. Si une modification ne touche que ces champs, elle est traitee comme un non-changement. Ce filtrage est applique a trois niveaux : enregistrement, lecture et restauration.

Borne memoire : Le tampon est limite en RAM. Au-dela de 10 000 entites ou 200 Mo, les snapshots sont flushed vers une table staging SQLite pour eviter les debordements memoire sur les editions massives.

Stockage delta

Le mode PostgreSQL historique sauvegardait la ligne entiere a chaque modification (row_to_json(OLD)). Sur une table a 50 colonnes, modifier un seul champ stockait 50 valeurs. Le mode local adopte une strategie differenciee :

OperationStrategieContenu stocke
DELETESnapshot completTous les attributs + geometrie WKB (l'entite disparait, tout est necessaire pour la recreer)
UPDATEDelta uniquementAnciennes valeurs des seuls attributs modifies + ancienne geometrie si elle a change
INSERTSnapshot completEtat de l'entite apres creation (pour permettre un "undo insert")

Gain de stockage

Avant (mode PG)
Snapshot complet

50 attributs stockes pour chaque UPDATE, meme si un seul a change. Volume eleve, bruit dans la preview.

Maintenant (mode local)
Delta intelligent

Seuls les attributs reellement modifies sont stockes. Reduction 10-50x du volume. La preview montre directement "ce qui a change".

Format du delta

Le champ attributes_json change de semantique selon l'operation :

DELETE : {"all_attributes": {"nom": "Dupont", "age": 42, "ville": "Paris"}}
UPDATE : {"changed_only": {"ville": {"old": "Paris", "new": "Lyon"}}}
INSERT : {"all_attributes": {"nom": "Martin", "age": 35, "ville": "Lyon"}}
Serialisation robuste : Chaque type QVariant (QString, int, double, QDate, QDateTime, QByteArray, bool, listes, maps, NULL) est converti en JSON de maniere reversible. Les BLOB sont encodes en base64 avec prefixe b64:. Les NaN et Infinity sont stockes comme null avec un flag.

Systeme d'identification

Le systeme utilise trois niveaux d'identification pour retrouver precisement chaque entite dans chaque couche de chaque projet.

Fingerprint de couche

Le fingerprint est la piece centrale du systeme d'identification. Il permet de distinguer deux couches qui portent le meme nom mais proviennent de sources differentes.

Comment il est calcule

Format : provider::source_normalisee

Type de coucheExemple de fingerprint
Shapefileogr::c:/data/parcelles/vegetation.shp
GeoPackageogr::c:/data/communes.gpkg|layername=batiments
PostGISpostgres::host=10.241.228.107 port=5432 dbname=AUVERGNE schema=rbal table=cables
SpatiaLitespatialite::c:/data/local.sqlite|layername=routes
GeoJSONogr::c:/export/zones.geojson
Deux couches "vegetation" dans deux dossiers differents ont des fingerprints differents car le chemin absolu est different. Le nom affiche dans QGIS n'est jamais utilise comme identifiant.

Force d'identification

Tous les formats n'offrent pas le meme niveau de fiabilite pour identifier une entite. RecoverLand qualifie chaque format avec une force d'identification.

ForceSignificationFormatsRisque de restauration
STRONGCle primaire stable, ne change pas entre les sessionsGeoPackage, SpatiaLite, PostgreSQL, Oracle, MSSQLMinimal
MEDIUMFID generalement stable mais peut changer apres certaines operationsShapefile, GeoJSON, FlatGeobufPossible si le fichier a ete recompacte ou re-exporte
WEAKPas de cle primaire, FID = numero de ligneCSV, texte delimitie, ExcelEleve : le numero de ligne change des qu'une ligne est ajoutee ou supprimee
NONEPas d'identifiant stable, donnees non persisteesCouche memoireRestauration impossible (capture a titre informatif uniquement)

Mode dual backend

RecoverLand supporte deux backends d'audit simultanement. Le BackendRouter decide automatiquement quel backend utiliser pour chaque couche.

Regles de routage

SituationBackend choisiRaison
Couche PostgreSQL avec triggers d'auditPostgreSQL legacyLes triggers serveur capturent deja les modifications
Couche PostgreSQL sans triggersSQLite localPas de capture serveur, le plugin prend le relais
GeoPackage, Shapefile, SpatiaLite, GeoJSONSQLite localFichiers locaux, pas de serveur
Couche non editable ou virtuelleAucunRefus explicite avec message
Pas de double audit : Une couche n'est jamais auditee par les deux backends en meme temps. Le routeur garantit qu'un seul backend est actif par couche et par session.

Derive de schema

Entre le moment ou une modification est enregistree et le moment ou l'utilisateur tente de la restaurer, la structure de la couche peut avoir change. Le module schema_drift.py detecte ces derives avant toute restauration.

Ce que le module compare

Chaque evenement stocke le schema des champs au moment de la capture (field_schema_json). Avant restauration, ce schema historise est compare au schema actuel de la couche :

Type de deriveImpactAction du plugin
Champ ajoute depuis la captureLe snapshot ne contient pas ce champLe champ est laisse a sa valeur par defaut
Champ supprime depuis la captureLe snapshot contient un champ qui n'existe plusLa valeur est ignoree avec avertissement
Type de champ modifieIncompatibilite de typeTentative de conversion, refus si impossible
Champ renommeAncien nom introuvableTraite comme champ supprime + champ ajoute
Rapport de compatibilite : Avant chaque restauration, un rapport explicite liste les champs compatibles, les champs manquants et les champs incompatibles. La restauration n'est jamais silencieusement partielle.

Integrite et recuperation

Le module integrity.py verifie la sante du journal au demarrage du plugin et apres tout incident.

Verifications au demarrage

VerificationAction en cas de probleme
PRAGMA integrity_checkNotification utilisateur si corruption detectee ; proposition de renommer le fichier et recreer un journal vide
Verification du fichier WALCheckpoint automatique pour consolider les ecritures en attente
Table schema_versionDetection de migration interrompue ; reprise ou alerte
Fichier de recuperation (recoverland_pending.json)Reintegration des evenements en attente perdus lors d'un crash precedent
Zero perte silencieuse : Si des evenements n'ont pas pu etre ecrits (crash, disque plein), le plugin les sauvegarde dans un fichier de recuperation et les reintegre au prochain demarrage. L'utilisateur est toujours informe.

Ecriture asynchrone

Les evenements sont ecrits dans SQLite par un thread dedie (WriteQueue). Le thread UI n'attend jamais la fin de l'ecriture. La queue utilise des lots de 500 evenements par transaction pour maximiser les performances.

Thread UI → queue.Queue → Writer thread → SQLite (batch 500)

Retention et purge

Le journal grossit au fil du temps. Le module retention.py offre trois mecanismes pour maitriser sa taille :

Par age
Purge par anciennete

Supprimer les evenements plus anciens qu'un seuil configurable (30, 90, 365 jours).

Par session
Purge par session

Supprimer tous les evenements d'une session d'edition specifique.

Compactage
VACUUM

Recuperer l'espace disque apres une purge. Reduit la taille physique du fichier SQLite.

Statistiques
Etat du journal

Nombre d'evenements, taille du fichier, age du plus ancien evenement. Visible dans la barre d'information.

Taille typique : Un evenement UPDATE delta (2 attributs modifies) occupe 200 a 500 octets. Un DELETE avec 20 attributs + geometrie point occupe 1 a 3 Ko. 10 000 modifications representent entre 5 et 20 Mo.

Cycle de vie d'un evenement

Edition QGIS → Snapshot (RAM) → Commit ? → Effet net → WriteQueue → SQLite

Si l'utilisateur fait un rollback au lieu d'un commit, le snapshot en RAM est simplement supprime. Rien n'est ecrit dans le journal.

Identification utilisateur

Chaque evenement porte un user_name resolu automatiquement par le plugin. Les sources sont testees dans l'ordre :

Config plugin → Variable env → os.getlogin() → Profil QGIS → "unknown"

La premiere source non vide est utilisee. Aucun evenement n'est cree avec un user_name vide.
