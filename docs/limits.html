RecoverLand : Limites

Limites et cas d'echec

Cette page documente honnetement les situations ou RecoverLand ne fonctionne pas, fonctionne partiellement, ou peut produire un resultat incorrect. Chaque cas est accompagne d'une explication de la cause et d'une recommandation pour l'eviter.

Pourquoi cette page existe : Un outil de recuperation de donnees doit etre transparent sur ce qu'il ne peut pas faire. Connaitre les limites evite les fausses certitudes.

Echelle de severite

NiveauSignification
CRITIQUEPerte de donnees ou restauration sur la mauvaise entite. L'utilisateur doit comprendre ce risque avant d'utiliser RecoverLand dans ce contexte.
MODERELe systeme fonctionne mais avec une fiabilite reduite. L'historique reste consultable mais la restauration automatique est deconseilllee.
FAIBLECas de bord rare. Impact limite a un scenario specifique.

Fichier deplace ou renomme

MODERE

Le probleme

Le fingerprint de couche utilise le chemin absolu du fichier source. Si le fichier est deplace, renomme ou copie ailleurs, le nouveau chemin produit un fingerprint different. L'historique enregistre sous l'ancien chemin n'est plus relie a la couche.

Consequence

L'historique existant reste intact et consultable (il n'est pas supprime). Mais RecoverLand ne sait pas que le fichier deplace est le meme que l'ancien. La recherche par couche ne montre pas l'historique ancien.

Comment l'eviter

  • Ne deplacez pas les fichiers de donnees apres avoir commence a les editer dans QGIS
  • Si vous devez reorganiser vos dossiers, faites-le avant de commencer le travail d'edition
  • Utilisez des chemins stables (pas de cles USB, pas de dossiers temporaires)

Lecteur reseau avec montage variable

MODERE

Le probleme

Si un fichier est accessible via Z:\partage\vegetation.shp sur un poste et via \\serveur\partage\vegetation.shp sur un autre, les deux chemins produisent des fingerprints differents bien que ce soit le meme fichier physique.

Meme probleme si la lettre du lecteur change : Z: un jour, Y: le lendemain.

Comment l'eviter

  • Utilisez toujours le meme chemin pour acceder au fichier (toujours la lettre de lecteur, ou toujours le chemin UNC)
  • Fixez la lettre du lecteur reseau pour qu'elle ne change pas entre les sessions

FID instable (shapefile, GeoJSON)

CRITIQUE

Le probleme

Le FID d'un shapefile est le numero d'enregistrement dans le .dbf. Ce numero peut changer si :

  • Le shapefile est re-exporte ("Enregistrer sous..." dans QGIS ou un autre outil)
  • Le fichier .dbf est compacte (Pack) apres des suppressions
  • Un script externe reorganise les enregistrements
Risque concret : Vous avez enregistre une modification sur l'entite FID=42 (un batiment). Le shapefile est re-exporte. Le FID 42 designe maintenant une route. Si vous restaurez, RecoverLand ecrit les attributs du batiment sur la route. Les donnees sont corrompues.

Comment l'eviter

  • Utilisez GeoPackage au lieu de Shapefile quand c'est possible
  • Si vous devez utiliser un shapefile, ne le re-exportez pas entre les captures et les restaurations
  • Avant de restaurer sur un shapefile, verifiez manuellement que le FID correspond bien a l'entite attendue

FID = -1 (entite en cours de creation)

FAIBLE

Le probleme

Pendant la creation d'une entite (avant le commit), QGIS attribue un FID temporaire de -1. Si le plugin capture un snapshot a ce moment, l'identite enregistree est {"fid": -1}, ce qui n'est pas unique.

Impact reel

Ce cas est rare car RecoverLand capture au moment du commit (afterCommitChanges), pas pendant l'edition. A ce stade, QGIS a deja attribue un FID definitif. Le risque est theorique.

Modifications hors QGIS

CRITIQUE

Le probleme

RecoverLand ecoute les signaux d'edition de QGIS. Si un fichier est modifie par un autre logiciel (ArcGIS, script Python, editeur de texte, ogr2ogr), RecoverLand ne detecte rien. Ces modifications ne sont pas dans le journal.

Comment l'eviter

Faites toutes vos modifications dans QGIS. Si vous devez utiliser un script, executez-le dans la console Python de QGIS en mode edition (startEditing / commitChanges) pour que les signaux soient emis.

Enregistrement desactive

CRITIQUE

Si l'utilisateur desactive le toggle d'enregistrement, aucune modification n'est capturee. RecoverLand ne peut pas recuperer ce qui n'a pas ete enregistre. Il n'y a pas de mode "rattrapage" apres coup.

Verifiez que l'enregistrement est actif (toggle vert) avant de commencer a editer. C'est la premiere chose a verifier si vous constatez que l'historique est vide.

Crash de QGIS

FAIBLE

Le probleme

Si QGIS plante pendant une session d'edition (avant le commit), les modifications en cours ne sont pas capturees car le commit n'a jamais eu lieu.

Si QGIS plante apres le commit mais avant que la WriteQueue ait ecrit dans SQLite, quelques evenements en file d'attente peuvent etre perdus. Le mode WAL de SQLite protege les evenements deja ecrits.

Impact reel

La WriteQueue ecrit par lots de 500 evenements ou a la fermeture. En usage normal (quelques modifications a la fois), les evenements sont ecrits rapidement. Le risque de perte est limite a quelques secondes d'activite avant le crash.

Mecanisme de recuperation

Au demarrage, le module integrity.py effectue plusieurs verifications :

  • PRAGMA integrity_check pour detecter une corruption du journal
  • Checkpoint WAL pour consolider les ecritures en attente
  • Recherche d'un fichier recoverland_pending.json contenant des evenements non ecrits lors du crash precedent
  • Reintegration automatique de ces evenements avec notification a l'utilisateur
Zero perte silencieuse : Si des evenements sont perdus malgre les mecanismes de recuperation, l'utilisateur est explicitement informe. Aucune perte n'est masquee.

Derive de schema

MODERE

Le probleme

Si la structure d'une couche change entre le moment de la capture et la restauration (champ ajoute, supprime, renomme, ou type modifie), la restauration peut echouer ou produire un resultat incomplet.

Ce que RecoverLand fait

Avant la restauration, le module schema_drift.py compare le schema sauvegarde avec le schema actuel et genere un rapport de compatibilite. Les champs manquants ou incompatibles sont signales.

Bonne pratique : Evitez de modifier la structure d'une couche (ajout/suppression de champs) entre les captures et les restaurations. Si c'est necessaire, consultez d'abord l'historique pour voir si des restaurations sont prevues.

Edition concurrente

FAIBLE

RecoverLand est concu pour un usage mono-poste. Si deux utilisateurs editent le meme fichier simultanement, chacun a son propre journal. La restauration par l'un peut ecraser les modifications de l'autre sans avertissement.

Ce n'est pas un scenario supporte. Pour l'edition collaborative, utilisez une base de donnees avec un systeme de verrouillage (PostGIS, par exemple).

Couche memoire

FAIBLE

Les couches memoire n'ont pas de fichier source persistant. RecoverLand capture les modifications a titre informatif, mais la restauration est impossible car la couche n'existe plus apres la fermeture de QGIS.

Taille du journal

FAIBLE

Le probleme

Le journal grossit indefiniment si la retention n'est pas configuree. Sur un projet avec des milliers de modifications quotidiennes, le fichier peut atteindre plusieurs centaines de Mo.

Impact

Les performances de recherche se degradent graduellement. SQLite reste performant jusqu'a plusieurs Go, mais les requetes deviennent plus lentes au-dela de 100 000 evenements.

Solution

Utilisez le module de retention (retention.py) pour purger les evenements anciens. Par exemple, conserver 90 jours d'historique et purger le reste.

Impact du stockage delta

Le mode local v3.0 utilise le stockage delta pour les UPDATE : seuls les attributs reellement modifies sont enregistres. Sur une table a 50 colonnes, modifier un seul champ stocke 1 valeur au lieu de 50. Cela reduit le volume du journal de 10 a 50 fois par rapport au mode PostgreSQL historique (qui stockait la ligne entiere).

Type d'evenementTaille estimee
UPDATE delta (2 attributs simples)200 a 500 octets
UPDATE delta (geometrie polygone simple)1 a 5 Ko
DELETE snapshot (20 attributs + point)1 a 3 Ko
DELETE snapshot (20 attributs + polygone complexe)10 a 100 Ko

Journal supprime

CRITIQUE

Si le fichier .recoverland/audit.db est supprime (manuellement, par un nettoyage de disque, ou par un antivirus), tout l'historique est perdu definitivement. RecoverLand creera un nouveau journal vide au prochain demarrage.

Ne supprimez jamais le dossier .recoverland dans le repertoire de votre projet QGIS, sauf si vous voulez explicitement effacer tout l'historique de modifications.

Projet QGIS deplace

MODERE

Si vous deplacez un projet QGIS (.qgz) dans un autre dossier, le project_fingerprint change. Le journal existant (qui est dans le dossier .recoverland/ de l'ancien emplacement) n'est plus trouve automatiquement.

Comment l'eviter

Si vous deplacez un projet, deplacez aussi le dossier .recoverland/ qui se trouve a cote du fichier .qgz. Le journal sera retrouve si le dossier .recoverland est present a cote du fichier projet.

Ce que RecoverLand n'est pas

RecoverLand n'est pas...Utilisez plutot...
Un systeme de versionnement (pas de branches, pas de merge)Git, GeoGig, ou un systeme de versionnement dedie
Un systeme de sauvegarde (ne copie pas les fichiers)Sauvegarde fichier classique, rsync, ou un NAS
Un systeme de collaboration (pas de synchronisation entre postes)PostGIS avec gestion de sessions, ou un SIG collaboratif
Un systeme de suivi en temps reel (pas de notification push)Triggers de base de donnees ou systeme d'audit au niveau SGBD
Un remplacement du Ctrl+Z (ne fonctionne qu'apres le commit)Le Ctrl+Z natif de QGIS pour les modifications non commitees

Ce que RecoverLand est

Un filet de securite local. Il enregistre automatiquement chaque modification validee dans un journal SQLite, et permet de retrouver et restaurer des etats anterieurs. Il protege contre les erreurs humaines courantes : suppression accidentelle, modification incorrecte d'attributs, ecrasement de geometries.

Il fonctionne mieux avec des formats a identification forte (GeoPackage, PostGIS), sur un poste de travail unique, avec des fichiers qui restent au meme emplacement.
